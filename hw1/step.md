  1. 先写图
  你的图建议用“邻接表”。

  核心思路：

  - 用一个字典保存每个节点的邻居
  - 邻居里同时保存边权，方便 UCS 用

  你至少需要这几个方法：

  - add_node(node)
  - add_edge(source, target, cost=1)
  - neighbors(node)
  - has_node(node)

  建议数据形式：

  {
      "A": [("B", 2), ("C", 1)],
      "B": [("D", 4)]
  }

  如果题目还要求“树”，最简单做法是：

  - 写一个 TreeNode
  - 每个节点有 value 和 children
  - 树可以单独展示，也可以转成图后复用搜索算法

  2. 写基础数据结构
  放到 src/data_structures.py。

  你只需要最基础功能：

  - Stack
      - push
      - pop
      - is_empty
  - Queue
      - enqueue
      - dequeue
      - is_empty
  - PriorityQueue
      - push(priority, item)
      - pop()
      - is_empty

  实现建议：

  - 栈：直接用 list
  - 队列：用 collections.deque
  - 优先队列：用 heapq

  3. 统一搜索结果格式
  在 search.py 里先设计好返回值，不然三个算法会越写越乱。

  建议每个搜索都返回：

  - found: 是否找到目标
  - path: 路径
  - cost: 路径总代价
  - explored_order: 访问顺序

  这样老师问“结果是什么”，你可以统一打印。

  4. 写路径还原函数
  三个搜索都建议共用一个辅助函数。

  思路：

  - 搜索时维护 parents
  - 例如 parents["E"] = "B" 表示 E 是从 B 来的
  - 找到目标后，从目标一路回溯到起点，再反转

  这一步非常关键，先写好再写算法。

  5. 写 DFS
  DFS 用栈。

  基本流程：

  1. 起点入栈
  2. 循环直到栈空
  3. 弹出一个节点
  4. 如果没访问过，就标记访问
  5. 如果它是目标，结束
  6. 把它的邻居压栈

  要点：

  - 一定要有 visited
  - 如果你想让输出顺序稳定，压栈时可以倒序处理邻居
  - DFS 不保证最优路径

  你可以先只保证“能找到目标”，别一开始就追求漂亮输出。

  6. 写 BFS
  BFS 用队列。

  基本流程和 DFS 很像，只是把“栈”换成“队列”。

  要点：

  - BFS 按层扩展
  - 如果每条边代价相同，BFS 找到的是步数最少路径
  - visited 最好在“入队时”就标记，不要等出队再标记，不然容易重复入队

  7. 写 UCS
  UCS 用优先队列。

  基本流程：

  1. 把起点以代价 0 放入优先队列
  2. 每次取出总代价最小的节点
  3. 用 best_cost 记录每个节点目前已知最小代价
  4. 如果发现更便宜的新路径，就更新并重新入队

  你需要维护：

  - parents
  - best_cost

  判断更新的核心条件：

  if new_cost < best_cost.get(neighbor, inf):
      ...

  要点：

  - UCS 和 BFS 最大区别是：BFS 比“步数”，UCS 比“总代价”
  - UCS 最容易写错的地方就是没有正确处理“更优路径更新”

  8. 构造示例图
  在 main.py 里手动建一个约 10 个节点的图。

  建议：

  - 节点：A, B, C, D, E, F, G, H, I, J
  - 加一些不同权重的边
  - 保证：
      - DFS 能找到一条路
      - BFS 找到“步数最短”
      - UCS 找到“代价最低”

  最好设计成：

  - BFS 的最短步数路径
  - UCS 的最低代价路径
    不一定完全一样
    这样你在报告里更好解释三者区别。

  9. 写打印函数
  不要把打印逻辑塞进算法里。

  单独写一个：

  - 输入算法名和结果
  - 输出 found/path/cost/explored_order

  这样代码结构会很干净。

  10. 最后写报告
  report.md 至少写这三部分：

  - 你实现了什么
  - 三种算法的区别
  - 完备性、最优性、时间复杂度、空间复杂度

  你可以按这个模板写：

  - DFS
      - 完备性：有限图且去重时完备
      - 空间复杂度：O(V)
  - BFS
      - 完备性：完备
      - 最优性：单位代价下最优
      - 时间复杂度：O(V + E)
      - 空间复杂度：O(V)
  - UCS
      - 完备性：正代价条件下完备
      - 最优性：最优
      - 时间复杂度：O((V + E) log V)
      - 空间复杂度：O(V)

  最容易出错的 5 个点

  - 忘记 visited，导致死循环
  - parents 没更新好，导致路径还原错误
  - BFS 在出队时才标记访问，导致重复入队
  - UCS 没有 best_cost，导致结果不最优
  - 图里只存邻居、不存权重，导致 UCS 没法写

  最稳妥的完成节奏

  1. 先写 Graph
  2. 写 Stack/Queue/PriorityQueue
  3. 写路径还原
  4. 先跑通 DFS
  5. 再改成 BFS
  6. 最后写 UCS
  7. 再补 main.py 和 report.md

› 我需要按照步骤，先用python实现一个图的数据结构，用邻接表来实现，请不要直接给我代码，给我合适的数据结构和详细的思路


• 目标

  你这一步只做一件事：把“图”表示出来，让后面的 DFS、BFS、UCS 都能基于它工作。

  最适合这次作业的方案是：
  用一个 Graph 类，内部用“邻接表”保存图。

  1. 先想清楚图里到底要存什么

  一个图最核心的是两类信息：

  - 有哪些节点
  - 每个节点能走到哪些邻居

  因为你后面还要写 UCS，所以边最好顺手带上“权重”。
  这样你的图结构一次设计好，DFS/BFS/UCS 都能复用。

  你内部可以这样理解：

  - 整体是一个字典
  - 字典的键：节点名，比如 A
  - 字典的值：这个节点的邻居列表
  - 邻居列表中的每一项：(邻居节点, 边权重)

  例如概念上可以长这样：

  - A 对应 [(B, 2), (C, 1)]
  - B 对应 [(D, 4)]

  这表示：

  - A 到 B 的代价是 2
  - A 到 C 的代价是 1
  - B 到 D 的代价是 4

  2. 为什么要用邻接表

  因为它特别适合搜索算法。

  你后面写 DFS/BFS/UCS 时，最常做的动作就是：

  - 取出当前节点
  - 找它的所有邻居
  - 决定下一个扩展谁

  邻接表正好就是“给我某个节点，我立刻能看到它的邻居”。

  相比之下，邻接矩阵更适合节点很少、查边频繁的情况。
  但你的作业只有十个左右节点，而且重点是搜索，邻接表更自然。

  3. 你的 Graph 类应该承担什么职责

  这个类不要一开始做太多事，只承担“存图”和“提供查询”这两类职责。

  你至少需要这些能力：

  - 创建一个空图
  - 添加节点
  - 添加边
  - 查询某个节点的邻居
  - 判断某个节点是否存在
  - 获取所有节点

  也就是说，你脑子里先定下这几个接口：

  - add_node
  - add_edge
  - neighbors
  - has_node
  - nodes

  先把这几个设计清楚，后面搜索算法会非常顺。

  4. 第一步：初始化图

  初始化时，你只需要准备一个空的邻接表。

  你的核心成员变量可以理解成：

  - 一个字典，开始时是空的

  此时图里还没有任何节点，也没有任何边。

  这一步的关键不是写多少逻辑，而是确定：

  - 图的唯一真实数据源就是这个邻接表
  - 后续所有节点和边信息都从这里读写

  5. 第二步：设计节点如何表示

  对于这次作业，节点最简单的表示方法就是字符串。

  例如：

  - A
  - B
  - C
  - J

  这样做的好处是：

  - 好打印
  - 好调试
  - 好写测试
  - 后面路径输出也很直观

  不要一开始把节点设计成复杂对象，除非老师明确要求。

  6. 第三步：设计 add_node 的行为

  add_node 的任务很单纯：

  - 如果这个节点还没出现过，就在邻接表里给它建立一个空邻居列表
  - 如果已经存在，就不要重复创建

  你要保证一个不变量：

  - 图中每个节点，在邻接表里都应该有一个键
  - 即使它暂时没有任何邻居，也要有一项空列表

  例如：

  - 加入节点 A 之后
  - 邻接表里应该能看到 A -> 空列表

  这件事很重要，因为以后可能会有“孤立节点”。

  7. 第四步：设计 add_edge 的行为

  这是整个图结构里最关键的方法。

  你需要先决定两个问题：

  - 你的图是有向图还是无向图
  - 是否允许边带权重

  我的建议是：

  - 默认按“有向图”设计
  - 但可以留一个参数表示是否双向
  - 权重默认值设为 1

  这样最灵活：

  - DFS/BFS 可以把所有边都当作代价 1
  - UCS 可以直接读取真实权重
  - 如果你想表示无向边，就加两次，或者在方法里自动补反向边

  add_edge 的合理思路是：

  1. 先确保起点存在
  2. 再确保终点存在
  3. 把 (终点, 权重) 加到起点的邻居列表里
  4. 如果是无向边，再把 (起点, 权重) 加到终点的邻居列表里

  这一步你要形成一个很清晰的认识：

  - 节点是“字典里的键”
  - 边是“某个键对应列表中的元素”

  8. 你需要提前决定：是否允许重复边

  这里有两种做法，你选一种就行。

  第一种：允许重复边
  优点是实现最简单。
  缺点是如果你不小心重复加边，邻居列表里会重复出现。

  第二种：不允许重复边
  优点是数据更干净。
  缺点是每次加边前要先检查是否已存在，逻辑会稍复杂。

  如果这是课程作业，我建议你先选第一种：

  - 先保证结构清楚、搜索能跑通
  - 后续如果老师要求严谨，再加“去重”逻辑

  9. 第五步：设计 neighbors 方法

  这个方法是给搜索算法用的，所以要非常稳定。

  它的作用是：

  - 给定一个节点
  - 返回它所有邻居以及对应权重

  例如你后面写搜索时会频繁做这种事：

  - 取当前节点 A
  - 拿到它的邻居列表
  - 遍历每个邻居继续扩展

  这里有一个设计细节值得注意：

  - 最好返回“邻居列表的副本”或者只读结果
  - 不要让外部直接改掉图内部存储

  不然以后可能出现这种问题：

  - 搜索算法不小心改了邻居列表
  - 原始图结构被污染

  10. 第六步：设计 has_node 和 nodes

  这两个方法是辅助型接口，但很实用。

  has_node 用来做什么：

  - 搜索开始前检查起点、终点是否存在
  - 避免因为访问不存在节点而报错

  nodes 用来做什么：

  - 打印图里有哪些节点
  - 调试时确认图是否构造正确
  - 方便你在 main.py 中展示结果

  这两个方法虽然简单，但能让你的主程序清晰很多。

  11. 你现在就要考虑“边权”统一问题

  虽然 DFS/BFS 不太关心权重，但 UCS 一定关心。
  所以你最好从一开始就统一规则：

  - 所有边都带权重
  - 如果你没特别指定，就默认是 1

  这样你后续不会出现“两套图结构”：

  - 一套给 DFS/BFS
  - 一套给 UCS

  那样会把作业写乱。

  统一成一种结构最省事：

  - DFS/BFS 忽略权重
  - UCS 使用权重

  12. 一个最推荐的图构造过程

  你可以按这个节奏自己动手：

  1. 先创建空图
  2. 手动加几个节点，比如 A B C D
  3. 再加几条边
  4. 打印整个邻接表，确认结构对不对
  5. 再尝试查询某个节点的邻居

  如果这一步你都能解释清楚，比如：

  - “A 的邻居是 B 和 C”
  - “B 的邻居是 D”
  - “C 暂时没有出边”

  那你的图结构就已经成功了。

  13. 你现在就可以给自己设一个验收标准

  写完图结构后，不急着写搜索，先问自己这几个问题：

  - 我能不能单独创建一个空图？
  - 我能不能添加一个孤立节点？
  - 我能不能添加一条带权边？
  - 我能不能查到某个节点的所有邻居？
  - 如果查一个不存在的节点，我希望返回什么？

  最后这个问题你要自己选风格：

  - 返回空列表
  - 或者抛出错误

  对作业来说，返回空列表通常更省事。

  14. 这一步最容易犯的错误

  你写图结构时，最常见的问题有这些：

  - 只加边，不先保证节点存在
  - 用列表存整个图，导致查邻居很不方便
  - 不存权重，后面 UCS 要重构
  - 邻居只存节点，不存边信息
  - 无向图时只加了一边，忘了反向边
  - 不小心让外部直接修改内部邻接表

  你只要提前防着这几个点，图这部分就不会出大问题。

  15. 你真正要记住的核心设计

  一句话概括：

  - 图是一个字典
  - 每个键是一个节点
  - 每个值是这个节点的邻居列表
  - 邻居列表里的元素是“邻居 + 权重”

  这就是你后面所有搜索算法的基础。
